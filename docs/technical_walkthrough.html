Technical Walkthrough

A detailed account of the data pipeline, terrain analysis, RF modeling, greedy node selection, and coverage mapping used to design the Wood County, WV minimal resilient mesh. All code is Python; all inputs are public-domain or open-licensed.

1. Data Sources

The pipeline uses five public or open-licensed datasets. No proprietary feeds or API keys are required for core operation.

USGS 3DEP 10m DEM is the primary elevation source, fetched via py3dep for the county bounding box and saved as a GeoTIFF in its native projected CRS. Every terrain operation in the pipeline runs against this single file.

Microsoft Building Footprints (West Virginia, ~37 MB) provides ML-extracted building polygons with substantially better rural coverage than OSM. The file is pre-filtered to the county bounding box on download, then clipped to the county boundary polygon. Buildings with footprint area under 15 m² are dropped as micro-sheds. 46,406 structures survive. If the Microsoft dataset is unreachable, the pipeline falls back to OSM buildings automatically.

TIGER/Line Roads supply the road network for visual context; only primary, secondary, trunk, and motorway classes are retained in the web bundle to reduce payload.

OpenStreetMap via osmnx provides critical infrastructure POIs: hospitals, clinics, fire stations, police, towers, masts, and water towers, as point geometries.

FEMA NFHL flood zones for Wood County (DFIRM_ID prefix 54107) are fetched from FEMA's ArcGIS REST API and used for visual context and site vulnerability assessment.

All outputs use EPSG:4326. Distance and area calculations use EPSG:32617 (UTM Zone 17N). DEM operations use the DEM's native CRS with explicit reprojection at every boundary crossing.

2. Pipeline Overview

The pipeline is a linear sequence of six Python modules, each reading from and writing to data/processed/. Any module can be re-run independently as long as its upstream outputs exist.

acquire.py downloads and preprocesses all raw datasets. terrain.py fetches the USGS 3DEP DEM via py3dep. analyze.py performs DEM local-maxima extraction, computes local prominence, and re-scores the pool by building density. select.py runs the LoS-aware greedy set-cover and writes node_placements.geojson. rf_model.py computes Friis link budgets, Fresnel-zone LoS for all node pairs, per-node viewsheds, and final building coverage. redundancy.py analyzes the mesh graph for single points of failure and average hop count. export.py bundles every processed layer into web_bundle.json for the frontend.

python -m pipeline.acquire
python -m pipeline.terrain
python -m pipeline.analyze
python -m pipeline.select
python -m pipeline.rf_model
python -m pipeline.redundancy
python -m pipeline.export

3. Terrain Analysis & Peak Extraction

The goal of analyze.py's _build_and_score() is to produce a geographically distributed pool of true local summits; not just the absolute highest points in the county. Sorting by absolute elevation and taking the top-N would cluster every candidate on the highest ridge system in the southeast, leaving the Parkersburg corridor and the river bluffs with no representation.

Local Maxima Detection

The DEM is read as a 32-bit float array. A pixel is flagged as a local peak if it equals the maximum value within its 5x5 neighborhood (via scipy.ndimage.maximum_filter), lies inside the county boundary mask, and is above a 150 m elevation floor. The 150 m floor discards Ohio River floodplain pixels, which are never viable siting candidates. At 10 m DEM resolution a 5x5 window spans a 50 m radius; tight enough to capture every meaningful hummock, including the ~220 m bluffs overlooking downtown Parkersburg.

neighborhood_max = maximum_filter(dem, size=5, mode="nearest")
is_peak = (dem == neighborhood_max) & poly_mask & (dem > 150)

Stratified Geographic Spread

The county's UTM bounding box is divided into an approximately 22x22 grid (n_side = ceil(sqrt(n_peaks))). Peaks are walked high-to-low and placed into their grid cell, subject to a 50 m minimum pairwise separation enforced within each cell and a per-cell cap to prevent any one cluster from consuming the budget. For n_peaks=500 over Wood County's ~800 km² envelope, each cell is roughly 1.7 km wide; approximately one candidate per cell on average. If the stratified pass exceeds the target count, candidates are trimmed by round-robin across cells, preserving geographic spread while respecting the count limit.

Local Prominence

For each candidate, local prominence is the candidate's elevation minus the mean elevation of all other candidates within a 1,500 m radius. This rewards sites that genuinely dominate their immediate neighborhood. A low-elevation bluff that rises 60 m above everything nearby scores higher on prominence than a high-elevation shoulder that is merely one of many equally-tall points on a plateau. Prominence is used as a secondary scoring term; it does not drive selection directly, but breaks ties in favor of sites with real terrain dominance.

prominence[i] = elev[i] - float(np.nanmean(elev[neighbors_within_1500m]))

4. Site Scoring

Scoring proceeds in two stages. The first is a provisional terrain-only score used to order the initial pool before building data is available. The second overwrites it with actual building counts.

Stage 1. Provisional Terrain Score

score = elev_m x 0.4 + prominence_m x 3.0

The prominence coefficient is deliberately elevated relative to absolute elevation to prevent the southeastern ridge system from monopolizing the top of the ranking. A 240 m bluff with 80 m of local prominence outscores a 380 m ridge shoulder with 10 m of prominence.

Stage 2. Building Density Rescore

After the pool is built, each candidate is rescored by the number of building centroids that fall within its radio horizon. A spatial bucketing optimization avoids computing a full 500 x 46,406 distance matrix: buildings are binned into 2 km grid cells, and each candidate only checks cells within ceil(horizon / 2000) + 1 cells of its own; reducing the inner loop from ~23 million operations to a few thousand per candidate.

Both factors are normalized to [0, 1] and combined:

score = (buildings_in_horizon / max_buildings) x 0.85
      + prominence_normalized x 0.15

The 85/15 split reflects the design goal: the mesh exists to serve people, and topographic prominence is a tie-breaker, not an objective in itself. This score is only used to produce the 200-candidate shortlist passed to select.py. The greedy selection algorithm ignores this score entirely and rebuilds per-candidate building-visibility sets from scratch using full DEM terrain profiles.

5. RF & Link Budget

rf_model.py models radio links using the Friis free-space path loss equation combined with a 4/3-Earth corrected radio horizon.

Friis Free-Space Path Loss

FSPL (dB) = 32.44 + 20*log10(d_km) + 20*log10(f_MHz)
max_loss   = Pt + Gt + Gr - Sens - Fade
           = 20 + 5 + 2 - (-137) - 10 = 154 dB
margin     = 154 - FSPL(d)

At SF12/BW125 the free-space budget of 154 dB closes at over 1,000 km on paper. That number is physically meaningless. The 4/3-Earth radio horizon bounds the practical range far sooner. For a 12 m mast:

d_horizon = sqrt(2 x (4/3) x 6,371,000 x 12) ~= 14,300 m

Node-to-node links use 12 m AGL at both ends, giving a mutual horizon of approximately 28.6 km maximum before Earth curvature becomes a hard obstacle. Candidate link evaluation is gated at MAX_LINK_KM = 40; pairs beyond 40 km are skipped without a DEM profile check. At 30 nodes the pairwise combination count is 435; most pairs are well under 40 km within a county-scale deployment.

6. Fresnel Zone & LoS Check

A naive LoS check asks only whether the straight line between two antenna endpoints clears terrain. This overestimates real connectivity by 2-3x because it ignores Earth curvature and the physical width of the radio beam. The pipeline applies both corrections at every profile sample point.

Earth Bulge (4/3-Earth Correction)

Atmospheric refraction causes radio waves to bend slightly downward, modeled by replacing the true Earth radius R with an effective radius k*R where k = 4/3. The effective bulge of this modified Earth at a point distance d1 from the transmitter along a total path D is:

bulge(d1) = (d1 x (D - d1)) / (2 x (4/3) x R_earth)

This bulge is added to the terrain elevation at every sample point before comparing to the LoS altitude. A pair that looks clear on a flat-Earth check may be blocked once the mid-path effective ground rises by several meters.

First Fresnel Zone Clearance

The first Fresnel zone is an ellipsoid around the direct LoS path. For reliable propagation, at least 60% of the first Fresnel radius must be clear at every point. The radius at a point d1 from the transmitter and d2 from the receiver is:

r1(d1, d2) = sqrt(lambda x d1 x d2 / (d1 + d2))

At 915 MHz, lambda = 0.327 m. At the midpoint of a 10 km link r1 ~= 28 m. A 60% clearance requirement demands the LoS path sit at least 17 m above any obstacle at that midpoint; substantially more than an endpoint-only check reveals. The required clearance altitude at each sample is then:

required  = ground_elev + bulge(d1) + 0.6 x r1(d1, d2)
clearance = los_altitude - required

A link is clear if clearance.min() >= 0 across all interior samples (endpoints excluded). NaN values in the profile; pixels outside the DEM raster extent; are filled by linear interpolation between valid neighbors. An all-NaN profile is treated as blocked.

Profile Sampling Depth

Node-to-node final links (rf_model.py) use 200 DEM samples per path. The building pre-screening in select.py uses 20 samples per building path, batched across all in-range buildings in a single sample_gen call, and 60 samples for inter-candidate connectivity checks. The coarser grids are acceptable for pre-screening but would not be used for final link validation.

7. Greedy Set-Cover Node Selection

select.py frames node placement as a weighted set-cover problem: the universe is the set of all building centroids in the county; each candidate site covers the subset of buildings reachable with clear LoS. The goal is to cover the most buildings with the fewest nodes, subject to mesh connectivity and geographic spacing constraints.

Visibility Matrix Construction

Before the greedy loop runs, the full N_candidates x N_buildings boolean visibility matrix is computed. For each candidate, buildings beyond the 14,300 m radio horizon are masked out immediately with a vectorized distance check; no DEM work needed for them. For remaining in-range buildings, all profile coordinates are computed at once and passed to a single sample_gen call, reading the DEM in one pass to avoid repeated I/O overhead. The Fresnel + bulge clearance check then runs over the returned elevation array. The result is a 200 x 46,406 boolean matrix held in memory. A separate N_candidates x N_candidates LoS matrix is built simultaneously using 60-sample profiles for each candidate pair.

Greedy Selection Loop

Round 1 selects the candidate with the highest raw building visibility count. This becomes the gateway anchor; the first node in the mesh. Subsequent rounds apply three constraints before evaluating candidates:

Connectivity constraint. A candidate may only be selected if at least one already-chosen node has a clear LoS path to it, verified via the pre-computed inter-candidate LoS matrix. This ensures the mesh remains a connected graph after every addition. A node that cannot talk to any existing node is useless regardless of how many buildings it can see.

Spacing constraint. A candidate must be at least 2,500 m from every already-chosen node in UTM distance. Without this, greedy selection piles nodes on top of the densest building cluster; the Parkersburg-Vienna corridor; and leaves rural ridges uncovered. The constraint forces geographic spread.

Marginal gain maximization. Among candidates satisfying both constraints, the one that adds the most previously uncovered buildings is selected. covered is a running boolean mask over all buildings; marginal gain is (visibility[candidate] & ~covered).sum().

connected_mask = np.any(cand_los[chosen], axis=0)  # sees any chosen node
spacing_mask   = min_dist_to_chosen >= 2500         # 2.5 km floor
gains          = (visibility & ~covered).sum(axis=1)

best = argmax(gains where connected_mask & spacing_mask)
covered |= visibility[best]

If no spaced+connected candidate adds new buildings, the loop falls back to the most-visible connected candidate to extend mesh reach even if marginal gain is zero. If no connected candidate exists at all, spacing is relaxed and the isolated pick is flagged in the output for review.

Gateway Designation

After all 30 nodes are selected, the three nodes with the highest raw building LoS count; not marginal gain, but total raw visibility; are designated gateways. These are the best candidates for internet backhaul connections because they have the most direct reach to population. This designation is a post-hoc label applied to an already topology-aware placement; the mesh connectivity does not depend on which nodes are gateways.

8. Viewshed & Coverage Mapping

After node placements are finalized, rf_model.py constructs a viewshed polygon for each node by shooting 1,440 radial rays (0.25 degree angular resolution) outward and finding the furthest point along each bearing that still satisfies 60% first Fresnel clearance to a 4 m rooftop receiver. The radial search walks outward in 150 m steps, sampling 40 DEM points per profile at each test distance, until clearance fails or the radio horizon is reached.

The 1,440 polygon vertices per node are stitched into a coverage polygon and saved to coverage_radii.geojson. The union of all 30 viewsheds is dissolved into a single combined footprint, then cleaned with a 150 m buffer-out / buffer-in operation to fill slivers between adjacent viewsheds, followed by 80 m simplification to remove vertex-count jitter from the ray-sampling boundary. The result; coverage_combined.geojson; is what renders as the cyan overlay on the map.

Building Coverage Count

Coverage percentage is not derived from the viewshed polygon area. It is computed by testing each of the 46,406 building centroids for actual LoS against every node individually, using the same Fresnel + bulge check used in selection. A building is counted as covered if at least one node has a clear LoS path to its centroid. Buildings are processed in chunks of 512 to keep memory flat; all profiles for a chunk are sampled in a single sample_gen call. The final result; 86.7% of buildings with clear LoS to at least one node; is written to building_coverage_summary.json and displayed in the map's live metrics panel.

9. Network Graph & Redundancy

redundancy.py builds a networkx.Graph from the selected nodes and clear LoS links, then analyzes it for structural weaknesses.

Single points of failure (SPoF) are identified by iteratively removing each node and testing whether the remaining graph is still connected. Any node whose removal disconnects the graph is a SPoF. The count is displayed in the map metrics panel. A well-designed mesh minimizes SPoFs through triangulated redundancy; every node should have at least two independent paths to the rest of the mesh.

Average hop count is computed from all-pairs shortest path lengths and indicates how many relay jumps a message must traverse on average between any two nodes in the network.

for node in list(g.nodes):
    h = g.copy()
    h.remove_node(node)
    if not nx.is_connected(h):
        single_points.append(node)

The network graph metadata; node count, edge count, connectivity, SPoF list, average hop count; is written to network_graph.json and consumed by both the map frontend and any downstream analysis.

10. Visualization & Export

export.py assembles all processed GeoJSON layers into a single web_bundle.json for the frontend. All layers are simplified with a 0.0005-degree tolerance via shapely.simplify to reduce file size, except the building footprints which are passed through unsimplified so individual structures render correctly.

The frontend uses Leaflet for tile rendering and GeoJSON layer management, and D3.js for data joins. The map is intentionally non-interactive; scroll wheel zoom, drag, and double-click zoom are all disabled; because it serves as a scrollytelling backdrop, not an exploratory tool. Map state is driven entirely by an IntersectionObserver watching the narrative steps. As each step scrolls into view at a 55% intersection threshold, activateStep() removes all current layers, re-adds the appropriate subset for that step, and updates the live metrics panel. The colophon step (data-step="10") returns early from activateStep without touching the map, preserving whatever state step 9 left behind.
